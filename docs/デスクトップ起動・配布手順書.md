# Desktop 起動・パッケージ手順書

## 目的

この手順書は、`CIPHER VAULT` の `Electron` デスクトップ版について、以下を再現できるようにまとめたものです。

- 開発環境で起動する
- ビルド済みデスクトップ版を起動する
- Windows 向け配布パッケージを作成する

## 起動パターン

### 1. 開発用 Web 版を起動する

フロントエンドと API を個別に動かして、ブラウザで確認する方法です。

```powershell
npm run dev
```

起動後のアクセス先:

- Web: `http://localhost:5173`
- API: `http://localhost:3001`

用途:

- UI の見た目確認
- Web 側の動作確認
- API 単体の挙動確認

### 2. 開発用 Electron 版を起動する

`Electron` ウィンドウでアプリを起動し、デスクトップ版として確認する方法です。

```powershell
npm run dev:desktop
```

このコマンドで行われること:

- `apps/api` をビルド
- `Vite` の開発サーバーを起動
- `Electron` を起動

用途:

- デスクトップアプリとしての見え方確認
- `preload + IPC` を含む実動作確認
- 変更しながらの反復開発

### 3. ビルド済み Electron 版を起動する

本番に近い構成で、ビルド済みアセットを使って起動する方法です。

まずビルドします。

```powershell
npm run build:desktop
```

次に起動します。

```powershell
npm run start:desktop
```

この方法では:

- `apps/web/dist`
- `apps/api/dist`
- `apps/electron/dist`

を使ってアプリが起動します。

用途:

- 配布前の最終動作確認
- 開発サーバーに依存しない起動確認

## パッケージ化手順

### 1. 配布物を作成する

Windows 向けの `installer` と `portable` を生成します。

```powershell
npm run package:desktop
```

このコマンドで行われること:

1. `npm run build:desktop`
2. `electron-builder --win nsis portable`
3. `dist-desktop/` へ配布物を出力

### 2. 出力物を確認する

生成先:

```text
dist-desktop/
```

主な出力物:

- `CIPHER VAULT-0.1.0-x64-setup.exe`
  Windows 向けインストーラー
- `CIPHER VAULT-0.1.0-x64-portable.exe`
  展開不要で起動できる portable 実行ファイル
- `CIPHER VAULT-0.1.0-x64.exe.blockmap`
  更新配信用の補助ファイル
- `latest.yml`
  更新配信用メタデータ
- `win-unpacked/`
  展開済みアプリ一式

### 3. アイコンについて

配布時のアプリアイコンは以下を参照します。

- `build/icon.ico`
- 元画像: `build/icon.png`

`electron-builder` の設定は `package.json` の `build` セクションにあります。

### 3.1 `setup` と `portable` の違い

- `setup`
  通常のインストーラー版です。一般ユーザー向けの標準配布形です。
- `portable`
  インストール不要でそのまま実行できる版です。検証用途や、インストール権限がない環境で使いやすいです。
- どちらもアプリデータの保存先は Electron の `userData` 配下です。
  `portable` であっても、実装上は exe と同じフォルダに DB を保存しません。

### 3.2 公開方法

- OSS として公開する場合は、`GitHub Releases` または公開ダウンロードページに `setup` と `portable` を掲載する形で問題ありません。
- 公開時の導線としては、一般ユーザー向けに `setup` を先に案内し、必要な人向けに `portable` を併記する形を推奨します。
- リポジトリ本文にはソースコード利用者向け手順、`Releases` には実行ファイル利用者向け案内を書くと分かりやすくなります。

### 4. データ保存先

デスクトップ版の DB は `apps/api/data` ではなく、実行ユーザーの `userData` 配下に保存されます。

Windows の例:

```text
C:\Users\<ユーザー名>\AppData\Roaming\CIPHER VAULT\data\password-manager.db
```

この保存先は Electron の `app.getPath("userData")` を基準に決まります。

## 日常の保守フロー

UI 改善や機能追加、軽微な保守を行うときは、基本的に次の流れで進めます。

1. 変更内容を整理する
2. 影響箇所を確認する
3. コードを修正する
4. `npm run dev:desktop` で実画面を確認する
5. `npm run build:desktop` でビルド確認する
6. 必要に応じて `npm run package:desktop` で配布物を確認する
7. 差分を見直す
8. コミットする
9. プッシュする

### 小さな変更の推奨フロー

```powershell
npm run dev:desktop
npm run build:desktop
git status
git add .
git commit -m "..."
git push
```

### 配布物に影響する変更の推奨フロー

```powershell
npm run dev:desktop
npm run build:desktop
npm run package:desktop
git status
git add .
git commit -m "..."
git push
```

### 補足

- UI 修正では `build` だけでなく `dev:desktop` での見た目確認を推奨します。
- アイコン、起動フロー、保存先、Electron 設定を触った場合は `package:desktop` まで確認したほうが安全です。
- 公開済み配布物を更新する場合は、`GitHub Releases` や配布ページへ新しい成果物を差し替えます。

## Codex を使う場合の運用フロー

このプロジェクトでは、`Codex` にコード修正を依頼し、手元で起動確認する流れが基本運用として適しています。

### 基本フロー

1. Codex に改善内容を依頼する
2. Codex がコード修正、関連ファイル更新、必要な設定変更を行う
3. 手元で `npm run dev:desktop` または `npm run build:desktop` を実行する
4. UI、動作、起動可否を確認する
5. 問題があれば再度 Codex に修正を依頼する
6. 問題がなければコミット、プッシュ、必要に応じて配布物を更新する

### 役割分担の目安

- Codex が担当しやすいもの
  実装、リファクタリング、設定変更、ドキュメント更新、エラー修正
- 人が確認したほうがよいもの
  UI の見た目、操作感、配布物としての違和感、最終的な採用判断

### 推奨コマンド

```powershell
# 見た目や挙動の確認
npm run dev:desktop

# 本番寄りのビルド確認
npm run build:desktop

# 配布物確認が必要な場合
npm run package:desktop
```

### 補足

- UI 修正や UX 改善は、Codex の修正後に必ず人が画面確認したほうが安全です。
- Electron の設定変更や配布関連修正は、`package:desktop` まで回して確認することを推奨します。

## トラブルシューティング

### `npm run start:desktop` で起動しない

以下を順に確認します。

1. 先に `npm run build:desktop` を実行しているか
2. `apps/electron/dist/main.js` が存在するか
3. 以前の `Electron` プロセスが残っていないか

### `npm run package:desktop` で失敗する

以下を確認します。

1. `npm install` をやり直す
2. `better-sqlite3` の再ビルドが通る環境か確認する
3. `build/icon.ico` が存在するか確認する
4. `npm run build:desktop` 単体が成功するか確認する

### アイコンが古いまま見える

Windows 側のアイコンキャッシュの影響で即時反映されないことがあります。
その場合は Explorer の再起動やサインアウト後に再確認します。
